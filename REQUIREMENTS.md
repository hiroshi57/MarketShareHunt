## 1. プロジェクト概要

### 目的
リアルな市場競争を体験できる、マルチプレイヤー対応の会社経営シミュレーターを作る。
プレイヤーが社員を採用し、案件を受注することで市場シェアを拡大する。

### ビジネスモデルのイメージ
Harness Engineering型（PLG + エンタープライズ二層構造）を組織モデルとして採用。

### 基本的な流れ

```
ホストが「次の月へ」を押すとターンが進む
↓
ニュース（NewsAPI + Perplexity API）＋季節要因で市場規模が変動
↓
市場に案件がランダム生成される（早い者勝ち）
↓
プレイヤーが社員を採用（役職・スキルに応じたコスト発生）
↓
スキルが合う案件を受注（難易度×期間で売上決定）
↓
ダッシュボードで損益・市場シェア・参入余地を可視化
```

## 2. 機能要件

### 必須機能（Must Have）

#### 採用システム
- 採用候補プール（40名）から社員を選んで採用
- 給与は役職・スキルレベルで変動（上位職ほど高い）
- 他プレイヤーに先に採用されると取れない（人材競合）
- 採用済み社員の一覧表示

#### 案件システム
- 季節要因（Q1〜Q4）＋ニュースAPI連携で案件をランダム生成
- 社員のスキルと案件要件のマッチングで受注可否が決まる
- 早い者勝ち（他プレイヤーに取られたら受注不可）
- 売上金額 = 案件の難易度×期間（月数）で決定
- 案件期間終了で収益も終了

#### 市場ダッシュボード
- 市場全体のパイ（市場規模）の可視化
- 自社シェア（%）の表示
- 参入余地（未受注の案件・領域）の可視化
- 競合他社のシェア表示

#### 損益ダッシュボード
- 総収益（売上）
- 総コスト（給与合計）
- 損益（利益 - コスト）
- ターン（月）ごとの推移グラフ

#### 時間システム
- ターン制（月単位）
- ホストが「次の月へ」ボタンを押すと全プレイヤーのターンが進む

#### マルチプレイヤー
- プレイヤー数制限なし
- 競争型（同じ市場でシェアを奪い合う）
- 終了条件なし（エンドレス）

#### ニュース・市場変動システム
- NewsAPI + Perplexity API でリアルニュースを取得
- ニュース内容を分析して案件の種類・量・市場規模に反映
- 季節要因（Q1〜Q4）と組み合わせて変動させる

### あると良い機能（Nice to Have）
- 部門別の損益内訳
- 景気指標の可視化（市場全体のトレンド）
- プレイヤーランキング

### 対象外（Out of Scope）
- 協力プレイ
- 実際の採用・案件管理業務への連携

## 3. 市場・案件の変動要因

| 要因 | 内容 | 影響 |
|------|------|------|
| 季節（Q1〜Q4） | 業務サイクルに応じた案件傾向 | 案件の種類・量が変動 |
| ニュース・時事 | NewsAPI + Perplexity API連携 | 市場規模・案件種別が変動 |
| 景気 | 上記2つの複合効果 | 市場全体のパイが拡大・縮小 |

### 季節イメージ
- Q1（1-3月）: 期末・決算・コスト削減系
- Q2（4-6月）: 新年度・組織改編・採用系
- Q3（7-9月）: 投資・MA・新規事業系
- Q4（10-12月）: 来期計画・予算消化系

## 5. データ定義

### 社員（Employee）

| フィールド | 内容 |
|-----------|------|
| 名前 | Haruto, Kaito など（40名） |
| 部門 | 01-経営企画部 〜 10-MA評価部 |
| 役職 | シニアストラテジストなど |
| スキル | 専門スキルリスト |
| 月次コスト | 役職・スキルレベルで変動 |
| 採用状態 | 未採用 / 採用済み（どの会社か） |

### 案件（Project）

| フィールド | 内容 |
|-----------|------|
| 案件名 | ニュース・季節から生成 |
| 必要スキル | マッチング対象 |
| 難易度 | 売上金額の係数 |
| 期間（月数） | 売上金額の係数 |
| 売上金額 | 難易度 × 期間で決定 |
| 発生要因 | 季節 / ニュース |
| 受注状態 | 未受注 / 受注済み（どの会社か） |

### 市場（Market）

| フィールド | 内容 |
|-----------|------|
| 市場規模 | ターンごとにニュース・季節で変動 |
| 領域別シェア | 部門・スキル別の市場構成 |
| 参入余地 | 未受注案件の合計 |
